# Architecture Overview - Tiny Tulip Coffee Console

## Executive Summary

This document provides a complete blueprint for building a production-ready web application to manage operations, marketing, and events at Tiny Tulip Coffee. The architecture is designed for:

- **Scalability**: PostgreSQL database with proper indexing and RLS
- **Security**: Row-level security, authenticated API access, encrypted credentials
- **Performance**: Real-time subscriptions, efficient queries, client-side caching
- **UX**: Mobile-responsive design, intuitive navigation, responsive layouts
- **Maintainability**: TypeScript for type safety, modular components, clear separation of concerns

## Data Flow Architecture

### Example: Creating an Event

```
1. User fills Event Form (React Component)
              ↓
2. Form validates with Zod schema
              ↓
3. Form data → eventService.createEvent()
              ↓
4. supabase.from('events').insert(data)
              ↓
5. API POST to Supabase REST endpoint
              ↓
6. Supabase Auth validates JWT token
              ↓
7. RLS policies check: is user 'admin' or 'manager'?
              ↓
8. INSERT trigger logs action to activity_log
              ↓
9. Response returned to client
              ↓
10. React Query cache invalidated
              ↓
11. UI updates with new event
              ↓
12. Realtime subscription notifies other users
```

### Example: Generating Checklist from Template

```
1. Staff selects event type in EventForm
              ↓
2. EventForm calls checklistService.generateFromTemplate(eventId, templateId)
              ↓
3. Calls supabase.rpc('create_checklist_from_template', {...})
              ↓
4. PostgreSQL function executes:
   - Create checklist row with event_id + template_id
   - Copy all template_items to checklist_items
   - Return checklist_id
              ↓
5. ChecklistManager component loads checklist
              ↓
6. Each item can be checked with sign-off capture
              ↓
7. Timestamp + user_id stored in checklist_signoffs table
              ↓
8. Completion percentage updates in real-time
```

## State Management Strategy

### Client State (React Component)
```typescript
// Local state for UI interactions
const [sidebarOpen, setSidebarOpen] = useState(false)
const [selectedEvent, setSelectedEvent] = useState<Event | null>(null)
const [formData, setFormData] = useState({...})
```

### Server State (React Query)
```typescript
// Cached server data with stale/refetch strategies
const { data: events } = useQuery(['events'], () => eventService.list())
const { data: inventory } = useQuery(['inventory'], () => inventoryService.list())
```

### Auth State (Supabase)
```typescript
// Managed by Supabase Auth
const { data: { session } } = await supabase.auth.getSession()
```

### Real-time State (Supabase Realtime)
```typescript
// Live subscriptions for collaborative features
supabase
  .from('checklists')
  .on('*', payload => updateChecklistUI(payload))
  .subscribe()
```

## Security Architecture

### Authentication
- Email/password authentication via Supabase Auth
- JWT token stored in browser (httpOnly not available in SPA)
- Token automatically included in API requests
- Automatic logout on token expiration

### Authorization (RLS)
Every table has RLS policies that enforce:
- Users can only see their own data (or data shared with them)
- Admins/managers have elevated access
- Activity log only viewable by admins
- Example policy:
  ```sql
  CREATE POLICY events_view ON events
    FOR SELECT USING (
      created_by = auth.uid() OR 
      EXISTS(SELECT 1 FROM users WHERE id = auth.uid() AND role IN ('admin', 'manager'))
    );
  ```

### Data Encryption
- HTTPS for all API calls (Supabase enforces this)
- Passwords hashed by Supabase Auth (bcrypt)
- Sensitive data not stored in localStorage
- Environment variables for secrets

### Input Validation
- Client-side: Zod schema validation
- Server-side: PostgreSQL constraints and triggers
- No direct SQL queries from client (uses Supabase RLS)

## Performance Optimization Strategy

### Database
- Indexes on frequently queried columns (date_start, status, event_type)
- Composite indexes for common filters
- Partitioning of large tables (activity_log) optional future
- Connection pooling via Supabase

### Frontend
- React Query caching with configurable stale time
- Code splitting by page (Vite dynamic imports)
- Lazy loading of modals and heavy components
- Image optimization (WebP, responsive sizes)
- CSS-in-JS replaced with Tailwind (no runtime overhead)

### Network
- Compression (gzip/brotli) automatic via Vercel
- CDN caching for static assets
- Real-time subscriptions for live features (vs polling)
- Batch API calls where possible

## Scalability Considerations

### For 1-100 Users
- Current architecture is sufficient
- Single database region (Supabase default)
- No special optimizations needed

### For 100-1000 Users
- Add database read replicas (Supabase Pro)
- Implement query result caching
- Consider edge functions for heavy computation
- Monitor and optimize slow queries

### For 1000+ Users
- Multi-region deployment (Supabase + Vercel Global CDN)
- Implement search indexing (PostgreSQL Full Text Search or Algolia)
- Cache frequently accessed data (Redis)
- Archival strategy for old events/logs
- Rate limiting on API endpoints
